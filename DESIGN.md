# Stock Agent Design System

更新時間：2026-07-11

這份文件是股票研究工作台的設計基線。它約束新 UI 與既有 UI 的驗收方式，不把現有畫面自動宣稱為完全符合；若既有樣式與本文件衝突，修改時要記錄取捨並補測試或截圖證據。

## 受眾與工作流

主要受眾是需要快速判斷研究資料、分析任務、報告品質與決策追蹤狀態的本機操作者。核心工作流依序是：

1. 輸入股票與選擇分析模式。
2. 確認任務、資料來源與模型狀態。
3. 閱讀報告、資料可信度與 evidence/conformance 品質訊號。
4. 進入追蹤、重跑、刷新或保存下一步操作。

介面優先服務掃讀、比較與可逆操作，不以行銷式展示或裝飾取代證據層級。

## 設計原則

- **證據先於動作**：資料新鮮度、來源異常、報告品質與重跑需求要在 CTA 附近可見。
- **決策語意先於內部代碼**：顯示「長線研究」或「部位決策」，不要把 `v1`、Agent 數或 provider 名稱當成主要文案。
- **一個視圖一個主任務**：每個工作區要有明確的主要 CTA，次要操作不與它競爭。
- **狀態不可只靠顏色**：每個品質或風險狀態都要有文字，必要時加圖示、標題或結構差異。
- **可逆且不自動升級風險**：查看、加入追蹤、選模式與開始分析分開；預填不等於自動送出。

## 字體與層級

目前正式前端基線使用 `Inter` 作為內文與控制項字體，`Outfit` 作為標題字體，來源設定在 `backend/static/styles/base.css`。

| 層級 | 用途 | 基線 |
| --- | --- | --- |
| Display | 首頁工作台標題 | 桌面約 `3.5rem`，手機依既有 breakpoint 收斂，不用 viewport 直接縮放 |
| Section | 工作區與面板標題 | `Outfit`、粗體、短句 |
| Body | 說明、欄位、主要資料 | 約 `1rem`，優先確保中文換行 |
| Supporting | metadata、時間、來源 | 約 `0.72rem` 到 `0.95rem`，不可承載唯一關鍵訊息 |

字距維持 `0`。長 ticker、公司名稱與模式文案必須允許換行或截斷提示，不能讓父容器被撐寬。

## 色彩與資料狀態

目前 token 以 `backend/static/styles/base.css` 的 `:root` 為基線：

| Token | 用途 |
| --- | --- |
| `--bg-dark` | 工作台背景 |
| `--text-primary` | 主要文字與高優先資料 |
| `--text-secondary` | 輔助說明與次要欄位 |
| `--text-muted` | 時間、來源與低優先 metadata |
| `--accent` | focus、連結與次要強調 |
| `--primary-action` | 主要可執行 CTA |
| `--glass-border` | 面板與資料區隔 |

資料狀態使用語意層：

- fresh/可用：文字明示資料新鮮或可用，不只使用綠色。
- partial/需留意：說明缺少哪些來源或限制，不只使用黃色。
- stale/error/blocked：說明過期、來源異常或阻擋原因，並提供可行動的刷新、重跑或等待路徑。
- 本機觀測與 provider 真實健康必須分開命名，不能用單一本機數字暗示外部服務完整健康。

裝飾性漸層、光暈或背景效果不得承載狀態、警示或可操作訊息；新 operational 元件優先使用穩定背景、邊框與文字層級。

## 版面與密度

- 首屏優先放輸入、模式、主要 CTA 與當前品質/任務狀態。
- 桌面工作台可使用寬版 grid；手機改為單欄堆疊，不以縮小字體解決空間問題。
- 新增重複資料卡時，圓角以 `8px` 以下為基線；既有 `glass-panel` 的大圓角只視為外層 shell，不應再包一層卡片。
- 控制項、表格欄位、模式選項與狀態徽章要有穩定的最小尺寸，避免 hover、長文案或 loading 造成版面位移。
- 每個區塊要有明確的留白與分組，避免在卡片裡再堆卡片；資料密集畫面優先使用欄位、分隔線與短標籤。
- 報告、資料可信度、追蹤績效與模型觀測不可混成同一個「總分」而失去來源脈絡。

## 可及性與互動

- 主要按鈕、連結、checkbox、select 與可點擊列表項目至少 `44px` 高或寬。
- 所有鍵盤可操作元素要有可見 `:focus-visible` 輪廓，且不能以 `outline: none` 取代可見 focus。
- 互動狀態同時提供 hover、focus、disabled、loading 與 error 的文字或結構差異。
- `prefers-reduced-motion: reduce` 時停用非必要動畫與平滑滾動。
- 確認、刪除、重跑等高影響操作要有清楚的確認語句；讀取與 mutation 的按鈕文案不可混淆。
- 使用熟悉圖示時要附 tooltip 或 accessible name；若沒有既有 icon library，不為裝飾手寫一套新的 icon 語言。

## Mobile 驗收

至少檢查 390px 與 768px 寬度，以及專案既有的 375px 小螢幕案例：

- 不得出現水平捲動或被截斷的主要 CTA。
- 模式名稱、公司名稱、警示與錯誤訊息必須能換行。
- 操作按鈕維持可觸控尺寸，資料卡改成單欄或可明確滑動的區域。
- 首屏要仍然看得到主要工作流的下一步，不用先穿過裝飾或大面積空白。

## 驗證門檻

UI 變更依風險選擇證據，不把單一綠燈外推成整體完成：

1. **靜態契約**：執行 `tests/test_static_history_filters.py`、對應文件契約與 Node `--check`。
2. **版面/互動**：執行 `scripts/visual_regression.sh`；若 Playwright 或瀏覽器不可用，要明確記錄 skip，而不是宣稱視覺通過。
3. **live runtime**：涉及 API、script order 或載入流程時，使用正式 `start_mac.command` 驗證 8080 的 HTML、asset、API 與 console/network 結果。
4. **人工檢查**：至少觀察桌面與手機 viewport，確認文字沒有重疊、主要 CTA 沒有被推出首屏、品質警示仍保留限制語句。
