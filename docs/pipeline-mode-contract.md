# 前後端模式契約

更新時間：2026-07-11

本文件是股票研究系統的模式對照基準，用來檢查前端選項、後端 pipeline、報告模板與使用者決策是否一致。若修改 `backend/pipeline_modes.py`、`backend/reporting/mode_templates.py` 或 `backend/static/ui_helpers.js`，必須同步檢查本文件與對應測試。

## 契約邊界

- 後端執行來源：`backend/pipeline_modes.py`
- Runtime 顯示 catalog：`backend/pipeline_mode_catalog.py`，唯讀 API 為 `/api/pipeline-modes`
- 報告模板來源：`backend/reporting/mode_templates.py`
- 前端初始 fallback：`backend/static/pipeline_mode_fallback.js`，由 `scripts/generate_pipeline_mode_fallback.py` 產生
- fallback CI 閘門：`scripts/generate_pipeline_mode_fallback.py --check`
- 前端模式 helper：`backend/static/ui_helpers.js`
- 前端 runtime merge：`backend/static/pipeline_mode_catalog.js`
- 驗證測試：`tests/test_report_mode_templates.py`、`tests/test_static_history_filters.py`、`tests/test_pipeline_mode_catalog.py`、`tests/test_pipeline_mode_metadata_sync.py`
- 文件與新整合範例一律使用 canonical pipeline ids：`v1` / `v2` / `v3` / `v4`。`mode_a`、`mode_b` 等 alias 只保留作為後端相容輸入，不應出現在新文件範例。

## 前後端模式漂移閘門

執行語意仍由 `backend/pipeline_modes.py` 維護，使用者可見的完整模式 catalog 則由 `backend/pipeline_mode_catalog.py` 組裝並透過 `/api/pipeline-modes` 提供。前端依序載入 generated `pipeline_mode_fallback.js`、`ui_helpers.js` 與 `pipeline_mode_catalog.js` 的 runtime merge：API 成功時採用 canonical catalog，API 失敗時保留可用的初始選項，不阻擋首頁操作。`tests/test_pipeline_mode_catalog.py` 驗證後端 catalog、generator/CI check 與 API，`tests/test_pipeline_mode_metadata_sync.py` 驗證生成 fallback、runtime merge，以及前後端已知欄位的漂移。這是 drift guard 加上 runtime canonical source 與可逆的 generated fallback，不把 API 或 fallback 任一層誤寫成完全消除所有來源的 single source。舊版 drift guard 曾記錄「仍需設計 API 或 build-time catalog」；本輪已完成 runtime API 與 build-time artifact，剩餘工作是持續維護生成命令與新增模式的 schema 相容性。

## 模式選擇速記

先問：我現在要做哪種決策？

- `v1`：要不要納入長線研究清單。
- `v2`：現在要進場、續抱、減碼或等待。
- `v3`：敘事是否過熱、是否需要避險或反向觀察。
- `v4`：未來 1-2 週是否有可執行事件窗口。
- `both`：同一檔股票需要三視角交叉檢查；它會產出 `v1`、`v2`、`v3` 三份獨立報告。

這個速記像分診台：先用使用者要做的決策分流，再回到下方表格檢查報告模板、摘要標題與證據要求。若只是寫文件或串接 API，使用 canonical id；若在程式內收到 alias，先交給後端 normalization。

## 模式選擇決策樹

1. 如果需要三視角交叉檢查，選 `both`，並預期產出 `v1`、`v2`、`v3` 三份報告。
2. 如果核心問題是 1-2 週事件窗口，選 `v4`，重點檢查催化日期、進場區間、目標與停損。
3. 如果核心問題是敘事過熱、泡沫或避險，選 `v3`，重點檢查泡沫證據鏈、反證條件與失效條件。
4. 如果核心問題是今天或本週要不要交易，選 `v2`，重點檢查進場、續抱、減碼或等待的風險報酬。
5. 如果核心問題是是否值得長線研究，選 `v1`，重點檢查基本面、估值區間、護城河與資料可信度。
6. 若仍不確定，先選 `v1` 建立基本面基準，再視報告結論補跑 `v2`、`v3` 或 `v4`。

這個順序先處理多報告需求與短效期限，因為它們最容易造成重跑成本；再處理風險框架與交易行動；最後回到長線研究基準。

## 契約矩陣速學卡

先問三題：

1. 有沒有改 `[投資建議]`、prompt、parser regex、`content_credibility` 或 template decision heading？
2. 有沒有改完整報告正文、Markdown/HTML 標題或 template 顯示文案？
3. 是不是只改前端 filter、preview、compare 或 rerun CTA？

這張速學卡把契約矩陣想成三道安檢通道：先判斷改動會進哪一道，再跑對應的最小測試命令。它不新增自動選測腳本；操作者仍要先判斷改動層級，避免把機器契約、報告呈現與前端顯示混成同一種風險。

| 通道 | 進入條件 | 最小測試命令 | 判讀 |
|---|---|---|---|
| 高顯著性機器契約通道 | 改 `[投資建議]`、`[/投資建議]`、`最終投資建議`、prompt、parser regex、`content_credibility` 或 template decision heading。 | `$(scripts/project_python.sh) -m pytest tests/test_content_credibility.py tests/test_report_preview.py tests/test_report_conformance.py tests/test_audit_rules.py tests/test_prompt_context_routing.py -q` | 綠燈代表核心機器契約未被已知測試打破；不得解讀為所有生成報告語意安全。 |
| 混合層報告呈現通道 | 改完整報告正文、Markdown/HTML 標題或 template 顯示文案，且使用者會直接閱讀。 | `$(scripts/project_python.sh) -m pytest tests/test_report_mode_templates.py tests/test_report_storage_integration.py tests/test_frontend_http_e2e.py -q` | 綠燈代表目前模板、儲存與 preview 路徑未回退；仍需人工檢查語氣是否降低權威感。 |
| 低顯著性顯示層通道 | 只改前端 filter、preview、compare 或 rerun CTA，且不碰 parser/template。 | `$(scripts/project_python.sh) -m pytest tests/test_static_history_filters.py tests/test_frontend_visual_optional.py -q` | 綠燈代表前端入口文字契約未回退；不得支持 parser、prompt 或 report template 契約詞替換。 |

## Quality Signal 與 Repair Queue CI Lane Map

P0/P1 系統優化後，報告品質不只由模式模板決定，也由 `content_credibility`、`report_conformance`、`evidence_exit_gate`、`data_trust`、`decision_freshness`、`report_quality_repair_queue`、`provider_impact`、`outcome_calibration`、`model_route_budget`、`notification_delivery` 與 `daily_decision_queue` 共同決定。任何修改若碰到這些 quality signal 或 operator queue，必須依風險選測試 lane，不得用單一綠燈外推整個報告系統安全。

| 修改類型 | 影響的契約 | 測試 lane | 判讀限制 |
|---|---|---|---|
| Report content quality gate：改 `content_credibility`、`report_conformance`、`evidence_exit_gate`、報告 renderer 或 conformance 規則。 | 報告是否可被採用、是否需要人工審核、是否能進入 repair queue。 | `$(scripts/project_python.sh) -m pytest tests/test_content_credibility.py tests/test_report_conformance.py tests/test_report_quality_repair_queue.py tests/test_docs_contract.py -q` | 綠燈只代表既有品質 gate contract 未回退，不代表所有生成報告內容都正確。 |
| Repair / provider impact lane：改 `report_quality_repair_queue`、`provider_impact`、`data_trust.reason_codes` 或 provider SLA impact mapping。 | blocked report、wait provider recovery、rerun/refresh action 的排序與阻擋條件。 | `$(scripts/project_python.sh) -m pytest tests/test_report_quality_repair_queue.py tests/test_provider_impact.py tests/test_daily_decision_dashboard.py -q` | provider warning 不等於 provider billing truth；核心/補充來源必須分開判讀。 |
| Outcome calibration lane：改 `outcome_calibration`、decision backtest detail mapping、miss attribution 或 quality signal join。 | backtest miss 如何歸因到 data quality、insufficient evidence、thesis error、timing error 或 unknown。 | `$(scripts/project_python.sh) -m pytest tests/test_outcome_calibration.py tests/test_daily_decision_dashboard.py tests/test_decision_tracking_workflow.py -q` | 回測綠燈不代表投資策略有效，只代表 attribution 與 payload contract 未回退。 |
| Model route / cost observability lane：改 `model_route_budget`、job telemetry、cache hit、retry storm 或 route warning。 | model/pipeline route 的 latency、retry、billable tokens 與 operator warning。 | `$(scripts/project_python.sh) -m pytest tests/test_model_route_budget.py tests/test_runtime_observability.py tests/test_import_boundaries.py::test_backend_python_modules_stay_split_below_threshold -q` | 沒有 verified price table 時，`estimated_cost_usd` 必須維持 null；不得填假精準成本。 |
| Daily decision queue lane：改 `daily_decision_queue`、`daily_decision_dashboard`、watchlist daily dashboard route、`notification_delivery` queue action 或 `actions` 相容 payload。 | blocked repair、provider wait、notification delivery repair、due backtest、rerun、model route warning、watchlist 與 screener 的每日排序。 | `$(scripts/project_python.sh) -m pytest tests/test_daily_decision_queue.py tests/test_daily_decision_dashboard.py tests/test_free_notification_plan.py tests/test_watchlist_service.py tests/test_decision_tracking_workflow.py -q` | top 5 只代表當日操作順序，不代表其他 secondary items 不重要；notification delivery repair 必須保持 `suppress_notification`，避免用壞掉的外部通道通知外部通道故障。 |
| Runtime/storage lane：改 `runtime_paths`、report artifact locator、report storage helper、decision tracking store 或 operational DB wiring。 | current state 是否仍來自 canonical DB，report artifact 是否仍透過 locator/storage helper 查找。 | `$(scripts/project_python.sh) -m pytest tests/test_runtime_paths.py tests/test_report_artifacts.py tests/test_import_boundaries.py -q` | 不得使用 `backend/cache/decision_tracking.sqlite3` 驗證 current state；不得手拼 `backend/output/<filename>`。 |

若一個 patch 同時碰到兩個以上 lane，必須跑所有相關 lane。若只改文件，至少跑 `tests/test_docs_contract.py`；若文件列出命令，文件命令必須與實際測試檔名一致。

## 契約矩陣操作流程

四步演算法：

1. 步驟 1：定位改動層級。先判斷這次改動碰到機器契約、報告呈現，還是只碰前端顯示。
2. 步驟 2：選擇安檢通道。用上方三題把改動放進高顯著性機器契約通道、混合層報告呈現通道或低顯著性顯示層通道。
3. 步驟 3：執行最小測試命令。先跑對應通道的命令；若改動跨兩個通道，跑兩組命令。
4. 步驟 4：記錄判讀與限制。在 PR、變更紀錄或 HCS 狀態表寫下綠燈代表什麼、不得解讀為什麼。

三個操作者情境：

| 情境 | 判斷方式 | 建議通道 |
|---|---|---|
| 情境 A：調整 parser、prompt、content credibility 或 decision heading | 只要影響 `[投資建議]`、content credibility、report conformance、prompt routing 或 template decision heading，就視為機器契約變更。 | 高顯著性機器契約通道 |
| 情境 B：改完整報告模板或正文標題 | 使用者會直接閱讀，且 report storage、mode template 或 HTTP preview 可能受影響。 | 混合層報告呈現通道 |
| 情境 C：只改前端顯示文案 | 只在 filter、preview、compare、rerun CTA 顯示，且不被 parser 或 template 讀取。 | 低顯著性顯示層通道 |

三條捷思規則：

1. `[` 或 `]` 出現契約詞就走高顯著性。
2. 使用者會直接閱讀的報告正文先走混合層。
3. 只在前端看得到且不被 parser 讀取才走低顯著性。

## 契約矩陣採用觀測板

最佳化目標：

1. 降低錯選測試命令：改動者能在改檔前選到正確通道，而不是事後靠失敗測試猜測。
2. 減少跨層改動漏跑測試：同時碰 parser/template/frontend 時，要明確跑多組命令。
3. 保留人工判斷責任：本觀測板不新增遙測或自動化蒐集，只定義人工檢查時該看的訊號。

可觀察假說：

| 假說 | 預期訊號 | 反證訊號 |
|---|---|---|
| 假說 1：四步流程會降低第一次選測摩擦 | 變更紀錄能在一段內說清楚改動層級、通道與命令。 | 改動者仍只貼一串測試命令，沒有說明為何選這組。 |
| 假說 2：三個操作者情境會降低錯選通道 | parser/prompt、報告模板、前端文案三種改動能各自對到正確通道。 | 完整報告模板改動只跑前端顯示層測試，或 parser 改動漏跑 conformance。 |
| 假說 3：三條捷思規則會減少低顯著性誤用 | 只有純前端顯示文案才使用低顯著性通道。 | 改 `[投資建議]`、prompt 或 template heading 時仍引用低顯著性通道。 |

採用訊號矩陣：

| 訊號 | 綠色 | 黃色 | 紅色 |
|---|---|---|---|
| 通道選擇 | 改動說明明確指向單一或多個通道，且命令相符。 | 通道大致正確，但沒有寫出跨層改動是否需要多跑命令。 | 改動層級與測試命令不相符。 |
| 測試判讀 | 說明綠燈代表什麼，也說明不得解讀為什麼。 | 只記錄測試通過，未寫推論限制。 | 把前端測試通過解讀為 parser/prompt safety。 |
| 後續行動 | 紅燈或黃燈時能回到正確通道與補跑命令。 | 只留下待辦，沒有指向通道。 | 忽略紅燈或用不相關測試替代。 |

## 契約矩陣案例模型

三類案例模型：

| 模型 | 典型改動 | 應對通道 | 核心風險 |
|---|---|---|---|
| 模型 A：高顯著性機器契約案例 | 調整 parser regex、prompt、`[投資建議]` 區塊或 template decision heading。 | 高顯著性機器契約通道 | 解析、稽核或 prompt routing 被打破，但前端測試仍可能通過。 |
| 模型 B：混合層報告呈現案例 | 改完整報告模板、Markdown/HTML 標題或使用者直接閱讀的正文文案。 | 混合層報告呈現通道 | 報告可讀語氣改善，但儲存、preview 或 mode template 合約回退。 |
| 模型 C：低顯著性顯示層案例 | 只改 filter、preview、compare、rerun CTA 等前端顯示文案。 | 低顯著性顯示層通道 | 純前端變更被過度升級，或反過來被誤用來支持 parser/template 變更。 |

代表性抽樣規則：

1. 每次契約相關變更至少對照一個案例模型。
2. 跨層改動同時抽樣兩個模型，並跑兩組對應命令。
3. 不以單一綠燈案例代表所有未來改動；案例只證明當次改動在該模型下的已知風險被檢查。

案例卡格式：

| 欄位 | 要填什麼 |
|---|---|
| 改動描述 | 這次改了哪些檔案、字串或模板。 |
| 選擇通道 | 對應哪個案例模型與安檢通道；若跨層，列出多個通道。 |
| 必跑命令 | 貼上本文件中對應的最小測試命令。 |
| 採用訊號 | 依採用訊號矩陣標成綠色、黃色或紅色，並寫下不得解讀為什麼。 |

## 契約矩陣比較與回饋設計

比較組設計：

| 組別 | 使用材料 | 比較目的 |
|---|---|---|
| 基準組：只使用速學卡與操作流程 | `契約矩陣速學卡`、`契約矩陣操作流程`。 | 看操作者能否只靠三題、四步流程與捷思規則選到正確通道。 |
| 介入組：使用案例模型與案例卡 | 加上 `契約矩陣案例模型`、`代表性抽樣規則`、`案例卡格式`。 | 看案例模型是否降低錯選通道與漏跑命令。 |

比較指標：

1. 錯選通道：改動層級與所選安檢通道不一致。
2. 漏跑命令：跨層改動只跑單一通道命令，或高顯著性改動漏跑 conformance/prompt/audit。
3. 判讀限制缺漏：只寫測試通過，未寫綠燈不得解讀為什麼。

介入方案：

1. 改檔前先填案例卡，再執行測試命令。
2. 跨層改動強制列出兩個模型，並列出兩組必跑命令。
3. 若採用訊號是黃色或紅色，回到比較指標補齊通道、命令或判讀限制。

訪談回饋題：

1. 你能否在 2 分鐘內選出通道？
2. 哪一條規則讓你猶豫？
3. 案例卡是否幫你發現漏跑命令或判讀限制？

本節只定義人工比較與回饋問題，不新增產品遙測、不新增自動化蒐集，也不把訪談回覆當成測試通過的替代證據。

## 契約矩陣觀察與複製準則

觀察記錄欄位：

| 欄位 | 記錄內容 |
|---|---|
| 變更案例 | 這次契約相關變更對應哪個案例模型，以及是否跨層。 |
| 實際選擇通道 | 改檔前實際選了哪些安檢通道，是否和案例模型一致。 |
| 實際執行命令 | 實際跑了哪些最小測試命令；若跨層，列出多組命令。 |
| 觀察結果 | 採用訊號是綠色、黃色或紅色；若非綠色，記錄補救行動。 |

複製檢查清單：

1. 同一案例模型：下一位操作者能否用相同模型判斷同類改動。
2. 同一必跑命令：同類改動是否能複製同一組最小測試命令。
3. 同一判讀限制：是否保留「綠燈不得解讀為所有生成報告語意安全」等限制。

可複製完成條件：

1. 下一位操作者不用讀完整 HCS 附件，也能從本文件找到通道、案例模型、命令與判讀限制。
2. 觀察記錄能指出通道選擇是否正確、命令是否漏跑、判讀限制是否缺漏。
3. 不把觀察紀錄當成測試替代品；完成仍需對應 pytest 或契約測試綠燈。

## 契約矩陣讀者路徑

三種受眾：

| 受眾 | 先看什麼 | 主要判斷 | 常見誤解 |
|---|---|---|---|
| 一般改文案者 | `契約矩陣速學卡` 與 `契約矩陣操作流程`。 | 判斷改動是否只在前端顯示層，並選擇低顯著性顯示層通道。 | 以為只改文案就完全沒有責任邊界；低顯著性不代表低責任。 |
| 報告模板維護者 | `契約矩陣案例模型` 與 `契約矩陣比較與回饋設計`。 | 判斷完整報告標題、HTML/Markdown 模板或使用者會閱讀的段落是否屬於混合層。 | 以為模板測試通過就代表使用者一定不會把報告當成交易指令。 |
| parser/prompt 維護者 | `契約矩陣觀察與複製準則` 與高顯著性機器契約案例。 | 判斷 `[投資建議]`、prompt、parser regex 或 decision heading 是否牽動機器契約。 | 以為文件列出通道後，工具會自動保證 parser、prompt 與報告語意安全。 |

閱讀順序：

1. 先讀速學卡：用三題判斷改動層級，先分出高顯著性、混合層或低顯著性通道。
2. 再用操作流程：照四步流程定位改動、選通道、跑最小測試命令、記錄判讀限制。
3. 最後填案例卡：把本次變更歸入案例模型，留下改動描述、選擇通道、必跑命令與採用訊號。

語意邊界：

1. 文件契約不是自動化保證；它要求操作者先做人工判斷，再用對應測試確認沒有已知回退。
2. 觀察紀錄不是測試替代品；它只能說明流程如何被使用，不能取代 pytest、parser、prompt 或報告模板契約測試。
3. 低顯著性不代表低責任；純前端顯示文案仍要避免把報告建議包裝成即時交易指令。

## 契約矩陣維護導覽

章節導覽：

| 順序 | 先看章節 | 用途 |
|---|---|---|
| 1 | 先判斷改動層級：`契約矩陣速學卡`、`契約矩陣操作流程`。 | 先判斷改動碰到機器契約、報告呈現，還是純前端顯示。 |
| 2 | 再選案例模型：`契約矩陣案例模型`、`契約矩陣比較與回饋設計`。 | 把改動放入高顯著性、混合層或低顯著性案例，並列出必跑命令。 |
| 3 | 最後確認模式對照：`模式對照`、`報告模板拆解`、`驗收標準`。 | 確認 pipeline id、報告模板、摘要標題、決策標題與使用者決策用途仍一致。 |

專業維護語氣：

1. 測試綠燈只證明已知契約未回退，不得宣稱投資語意安全、使用者一定理解、或所有歷史報告同步改善。
2. 跨層改動需列出多組命令，並分別說明每組命令保護的契約面。
3. 若證據只來自文件或觀察紀錄，應說「已記錄維護判斷」，不得說「已驗證 runtime 行為」。

核心論點：

1. 契約矩陣的目的不是自動化選測，而是讓操作者先保留人工判斷，再用最小測試驗證。
2. 任何改動碰到 parser/prompt/template，優先視為契約變更，再依實際範圍決定是否還要補跑報告呈現或前端顯示測試。
3. 純前端文案可以走低顯著性通道，但仍要維持報告層級語氣，不把報告建議寫成即時交易指令。

## 契約矩陣一頁摘要

短版摘要：

1. 先看是否碰 parser/prompt/template：若碰到 parser、prompt、`[投資建議]` 或 template decision heading，先走高顯著性機器契約通道。
2. 再看使用者是否會直接閱讀：若改完整報告正文、Markdown/HTML 標題或 template 顯示文案，走混合層報告呈現通道。
3. 最後看是否只在前端顯示：若只改 filter、preview、compare 或 rerun CTA，且不被 parser/template 讀取，才走低顯著性顯示層通道。

建議表達：

| 要說明的事 | 建議句型 |
|---|---|
| 通道 | 我選擇的通道是：高顯著性機器契約 / 混合層報告呈現 / 低顯著性顯示層，因為本次改動碰到的層級是... |
| 命令 | 我已執行的命令是：`pytest ...`，它保護的契約面是... |
| 限制 | 不得解讀為：所有生成報告語意安全、投資判斷正確、使用者一定理解，或未列出的 runtime 路徑已被驗證。 |

媒介取捨：

1. 文字與表格優先，因為本契約要讓維護者複製判斷、命令與限制，不是呈現產品儀表板。
2. 暫不新增圖像或多媒體，避免圖像把人工判斷包成自動流程，也避免讓操作者跳過文字中的限制條件。
3. 若未來真的需要圖像，必須同時保留文字版通道、命令與不得解讀為的限制。

## 契約矩陣倫理邊界

倫理底線：

1. 不得把測試綠燈寫成投資建議安全；測試只證明指定契約未被已知案例打破。
2. 不得把責任轉嫁給工具或文件；契約矩陣協助維護者判斷，但最終仍要由改動者說明通道、命令與限制。
3. 不得用低顯著性通道淡化使用者風險；只要文案可能讓使用者把報告當成即時交易指令，就要升級判斷。

必要時要說不：

1. 缺少 parser/prompt/template 證據時不可合併高顯著性改動。
2. 若報告文案可能被讀成交易指令，先補責任邊界，再談是否能走低顯著性通道。
3. 若跨層改動只跑單一測試命令，先要求補跑或拆分改動，不用綠燈包裝成已完成。

倫理判斷：

| 判斷項 | 允許發布的敘述 | 禁止發布的敘述 |
|---|---|---|
| 測試結果 | 已執行對應通道命令，已知契約未回退。 | 測試通過，所以投資語意安全。 |
| 文件證據 | 已記錄通道、命令與不得解讀為的限制。 | 文件已寫，所以工具會自動選對測試。 |
| 低顯著性 | 本次只影響前端顯示層，且未碰 parser/template。 | 低顯著性代表沒有使用者風險。 |

升級條件：

1. 從低顯著性升級為混合層：前端文案若會被使用者直接理解為報告正文、交易指令或責任聲明，就補跑報告呈現通道。
2. 從混合層升級為高顯著性：若改動碰到 parser/prompt/template、`[投資建議]` 或 decision heading，就補跑高顯著性機器契約通道。
3. 從文件判斷升級為 runtime 驗證：若宣稱包含實際執行行為、生成報告或 UI 呈現，必須提供對應測試或人工驗收證據。

## 契約矩陣系統風險邊界

複雜因果圖譜：

| 局部證據 | 可能造成的錯誤推論 | 系統邊界 |
|---|---|---|
| parser/prompt/template 測試通過 | 局部測試綠燈可能仍產生使用者誤解。 | 測試綠燈只覆蓋指定契約，不代表報告語氣、使用者行為或所有 runtime 輸出安全。 |
| 文件與觀察紀錄完整 | 文件紀錄可能降低漏跑測試但不保證採用。 | 需要實際改動紀錄、測試命令與 review 判讀，才能說明流程被使用。 |
| 前端顯示層語氣降低權威感 | 前端語氣改善可能與完整報告正文仍不一致。 | 前端測試不能替代報告模板、儲存、preview 或使用者閱讀情境檢查。 |

湧現風險：

1. 多個低顯著性改動累積成高風險：單次只改前端文案，但多次累積後可能改變整體使用者責任感。
2. 跨模式文案一致但責任邊界變模糊：所有模式看起來一致，仍可能讓交易、逆勢與長線研究的風險差異被抹平。
3. 觀察紀錄增加但實際驗證減少：文件填寫更完整，反而讓維護者誤以為不用跑對應測試。

分析層次：

| 層次 | 可用證據 | 不可替代的上一層或下一層 |
|---|---|---|
| 文件層 | 契約矩陣、案例卡、觀察紀錄、HCS 狀態。 | 不能替代測試層或 runtime 層。 |
| 測試層 | pytest、parser/prompt/template/front-end static tests。 | 不能替代 runtime 層或使用者行為層。 |
| runtime 層 | 實際生成報告、preview、HTTP 流程、UI 操作結果。 | 不能替代使用者行為層，也不能回推所有未測模板安全。 |
| 使用者行為層 | 真實操作者是否誤解、漏跑、過度採用報告建議。 | 不能用文件層或測試層直接宣稱已驗證。 |

不得用下一層證據替代上一層證據，也不得用上一層證據替代下一層證據；每次宣稱都要說清楚自己位在哪一層。

## 契約矩陣系統關係圖

維護網絡：

| 節點 | 連到哪裡 | 主要風險 |
|---|---|---|
| 前端顯示層 | 使用者判讀、測試矩陣、文件層。 | 顯示語氣改善但未同步報告正文或 parser/prompt 契約。 |
| 報告模板層 | runtime 層、測試矩陣、使用者判讀。 | 模板文案改動同時影響報告閱讀、preview 與儲存流程。 |
| parser/prompt 層 | 測試矩陣、runtime 層、報告模板層。 | 機器契約改動若只用前端測試驗證，會漏掉抽取、稽核與 prompt routing 風險。 |
| 測試矩陣 | 文件層、parser/prompt 層、報告模板層、前端顯示層。 | 測試通過被誤解成使用者行為安全或投資語意安全。 |
| 使用者判讀 | 前端顯示層、報告模板層、倫理邊界。 | 使用者可能把報告建議、比較結果或重跑提示當成交易指令。 |

系統動力學：

1. 語氣改善降低權威感但可能增加契約漂移：前端持續降權威語氣時，需同步檢查報告模板與 parser/prompt 是否仍使用必要契約詞。
2. 更多觀察紀錄降低漏跑但可能增加形式化：案例卡與觀察紀錄越完整，越要提醒仍需實際測試命令。
3. 更嚴格升級條件降低錯放但可能增加維護成本：高風險改動更安全，但低風險文案改善可能被過度阻擋，需保留低顯著性通道。

系統圖像：

1. 改動先定位層級：先標出前端顯示、報告模板、parser/prompt、文件或測試矩陣被改到哪一層。
2. 證據再對齊層次：文件層、測試層、runtime 層與使用者行為層的證據不可互相替代。
3. 宣稱最後受倫理邊界限制：完成敘述必須保留不得解讀為，不能把測試綠燈寫成投資語意安全。

## 契約矩陣 review 對話

補證據協商：

| 對話步驟 | 建議句型 | 目的 |
|---|---|---|
| 先承認改動目的 | 我理解這次想降低維護摩擦或改善使用者語氣，這個方向是合理的。 | 先承認改動的正當目的，降低 review 被理解成純阻擋。 |
| 再指出缺少的證據層 | 目前缺的是文件層、測試層、runtime 層或使用者行為層中的哪一層證據；不能用其他層級替代。 | 把爭論從立場轉回可補的證據層。 |
| 最後提出最小補證據路徑 | 最小補證據路徑是補跑對應測試命令、填案例卡，或拆成低顯著性與高顯著性兩個改動。 | 讓下一步可執行，而不是只留下抽象反對。 |

說服原則：

1. 把補跑命令說成降低錯放風險，而不是說成 review 的額外要求。
2. 把升級通道說成保護 parser/prompt/template，而不是對低風險文案改善的否定。
3. 把拆分改動說成降低 review 成本，而不是要求操作者重做整個 patch。

形塑行為：

1. 預設使用一頁摘要句型：回報通道、命令與不得解讀為，讓完成敘述自然保留限制。
2. 跨層改動預設填案例卡：只要同時碰前端顯示、報告模板或 parser/prompt，就先記錄改動描述、選擇通道與必跑命令。
3. 紅色或黃色採用訊號不得合併：紅色先補證據或拆分；黃色先補判讀限制，不能用綠燈語氣通過。

## 契約矩陣 review 防從眾檢查

防從眾檢查：

1. 不得用多數人同意取代證據層：即使 review 參與者都同意，也要說清楚文件層、測試層、runtime 層或使用者行為層的證據在哪裡。
2. 不得用前例綠燈取代本次改動層級：上一個低顯著性文案改動通過，不代表本次 parser/prompt/template 或報告正文改動也能走低顯著性通道。
3. 不得用測試全綠取代不得解讀為：所有指定測試通過，仍要保留限制句，不能說成投資語意安全或使用者一定理解。

差異保留：

1. 高顯著性、混合層、低顯著性不得合併敘述；完成回報要分別列出通道、命令與限制。
2. 長線、交易、逆勢、短線模式要保留不同責任邊界；不能因章節格式一致，就把決策用途與使用者風險寫成同一種語氣。
3. 文件層、測試層、runtime 層與使用者行為層要分開回報；其中一層有證據，不代表其他層已驗證。

情緒智商：

1. 先命名壓力：若時程、回歸失敗或 review 疲勞正在推動快速合併，先把壓力寫進回報。
2. 再回到最小補證據路徑：只要求補最小命令、最小案例卡或最小拆分，不把情緒壓力擴大成全面返工。
3. 最後用限制句收尾：即使為了降低維護成本採取最小路徑，也要說明不得解讀為哪些安全宣稱。

## 契約矩陣 review 責任分工

領導原則：

1. 主責先宣告改動層級：每個契約相關 patch 都要先說明自己屬於高顯著性、混合層或低顯著性，不把分類責任丟給 reviewer 猜。
2. review 主導者必須要求升級通道：只要改動碰到 parser/prompt/template、完整報告正文或使用者可能誤讀的責任聲明，就要主動要求升級或拆分。
3. 完成敘述必須保留不得解讀為：無論誰主導 review，最後回報都要說明測試綠燈不得解讀成投資語意安全或使用者行為已驗證。

權力動態：

1. 不得用職位或資深度取代證據：資深維護者同意、產品壓力或合併權限，都不能替代通道、命令與限制。
2. 低權限操作者可以引用契約矩陣要求補證據：任何人只要指出證據層缺口，都可以要求補跑命令、填案例卡或拆分改動。
3. 高權限操作者不得覆蓋紅色或黃色採用訊號：若採用訊號仍是紅色或黃色，必須先補證據、補限制或拆分，不能用權限直接改成綠色。

責任：

1. 改動者負責描述改動層級：改動者要說清楚碰到前端顯示、報告模板、parser/prompt、文件或測試矩陣的哪一層。
2. reviewer 負責核對通道與命令：reviewer 要檢查通道選擇是否和改動層級一致，並核對最小測試命令是否足夠。
3. 合併者負責確認限制句存在：合併者在接受改動前，要確認完成敘述包含不得解讀為與剩餘風險。

## 契約矩陣 review 自我稽核與收尾策略

自我覺察：

1. 契約矩陣不是自動化審核器：它只協助維護者說清楚改動層級、測試命令與限制，不替人判斷所有風險。
2. 規則變多可能增加官僚成本：若操作者為了填表而不再理解通道差異，矩陣本身就變成新風險。
3. 低顯著性顯示層不得被迫跑高顯著性全矩陣：只改前端 label、filter、preview 或 CTA 且不碰 parser/template 時，應保留輕量通道。

制定策略：

1. 先選最小足夠路徑：改動前先判斷層級，再選最小命令組與最小案例卡，不預設全矩陣。
2. 高風險升級、低風險保留輕量通道：碰 parser/prompt/template 或完整報告正文就升級；純顯示層仍走低顯著性。
3. 第 2 輪互動思考收尾條件：完成倫理邊界、系統風險、系統關係、review 對話、防從眾、責任分工與自我稽核。

收尾聲明：

1. 下一輪入口是第 3 輪批判思考：接下來應重新拆解契約矩陣是否過重、是否仍有最高風險缺口。
2. 不得宣稱 HCS Plus 完成：第 2 輪互動思考完成只代表本輪互動思考 20/20 收尾，仍需第 3 輪全習慣輪巡與後續綜合優化。

## 契約矩陣第 3 輪問題雷達

重新拆解：

| 問題 | 目前症狀 | 判斷 |
|---|---|---|
| 矩陣過重 | 維護者需要讀多個章節才知道通道、命令與限制句。 | 應把日常入口壓回一頁摘要，完整矩陣只留給高風險或跨層改動。 |
| 維護者是否能在 2 分鐘內選到通道 | 若只改前端 label 也要掃完整矩陣，低風險改動會被流程拖慢。 | 低顯著性顯示層需要保留輕量路徑，且完成敘述仍要保留限制句。 |
| 低顯著性是否被高顯著性流程拖慢 | 高風險 parser/prompt/template 護欄可能外溢到純前端顯示文案。 | 文件要明確說出何時不用完整矩陣，避免把安全流程變成官僚成本。 |
| 責任分工是否讓限制句真的出現 | 文件已分配改動者、reviewer、合併者責任，但尚未驗證完成敘述是否真的包含限制句。 | 第 3 輪需把責任分工轉成可檢查的回報格式或 review checklist。 |

關鍵問題：

1. 哪個規則可以被一頁摘要取代，而不降低 parser/prompt/template 的高顯著性保護？
2. 哪個情境必須保留完整矩陣，例如跨層改動、報告正文改動或改動 `[投資建議]` 契約詞？
3. 哪個證據層仍然沒有 runtime 驗證，只停在文件層、測試層或人工 review 層？

差距分析：

| 已完成 | 仍缺口 | 最小下一步 |
|---|---|---|
| 已完成速學卡、一頁摘要、三通道測試命令與 review 責任分工。 | 仍缺口是日常改動者可能不知道先看一頁摘要，或不知道何時必須升級完整矩陣。 | 在下一批把變數、偏誤與降低偏誤規則收斂成「一頁摘要優先、跨層改動升級」的判斷護欄。 |
| 已完成倫理、系統風險、防從眾與限制句要求。 | 仍缺口是限制句目前靠人記得寫，尚未有 runtime 或 PR 回報格式驗證。 | 先在文件契約中定義完成回報欄位，再評估是否需要自動檢查。 |
| 已完成低顯著性輕量通道。 | 仍缺口是輕量通道可能被誤用來支持 parser/prompt/template 改動。 | 下一批檢查哪些變數會造成誤用，並把偏誤降低規則寫進摘要與案例卡。 |

## 契約矩陣第 3 輪變數與偏誤降低護欄

變數分析：

| 變數 | 容易造成的誤判 | 判斷護欄 |
|---|---|---|
| 改動層級 | 把純前端 label 誤升級成高顯著性全矩陣，或把 parser/prompt/template 改動誤降級成只改文案。 | 完成回報先標註觸碰層級：前端顯示層、報告模板層、parser/prompt 層、文件層或測試層。 |
| 證據層 | 把文件契約、人工 review 或指定測試綠燈誤讀成 runtime 驗證。 | 分開列出文件層、測試層、runtime 層與使用者行為層的已驗證與未驗證事項。 |
| 可逆性 | 把跨層大改當成單一低風險 patch，導致回退成本被低估。 | 只要同時碰兩個以上層級，先拆分或升級完整矩陣；不能用低顯著性通道包住跨層改動。 |
| 時程壓力 | 為了趕合併省略限制句、案例卡或補跑命令。 | 黃色或紅色採用訊號不得因時程壓力改成綠色；只能補證據、補限制句或拆分。 |

偏誤辨識：

1. 過度升級偏誤：低顯著性前端文案被迫跑高顯著性全矩陣，導致小改動變慢。
2. 過度降級偏誤：parser/prompt/template 或完整報告正文改動被包裝成只改文案，漏掉機器契約風險。
3. 工具化幻覺：誤以為文件矩陣會自動判斷改動層級、阻擋合併或驗證 runtime 行為。
4. 綠燈擴張偏誤：把指定測試綠燈擴張成所有生成報告語意安全、使用者行為已驗證或投資判斷正確。

偏誤降低：

1. 一頁摘要優先：低風險改動先用一頁摘要選通道，只在觸發高風險變數時展開完整矩陣。
2. 跨層改動升級：只要同時碰前端顯示、報告模板、parser/prompt、文件或測試矩陣中的兩層以上，就升級完整矩陣或拆成多個 patch。
3. 證據分層回報：完成敘述必須列出文件層、測試層、runtime 層與使用者行為層哪些已驗證、哪些未驗證。
4. 限制句必填：每次引用測試綠燈都要寫出不得解讀為，例如不得解讀成投資語意安全或所有 runtime 路徑已驗證。
5. 案例卡觸發：跨層改動、黃色採用訊號或紅色採用訊號必須填案例卡；純低顯著性顯示層可保留輕量回報。

## 契約矩陣第 3 輪分流決策與效用校準

決策樹：

1. 只碰前端顯示層，且不被 parser/template 讀取：使用一頁摘要與低顯著性命令，完成回報保留限制句即可。
2. 碰 parser/prompt/template、`[投資建議]` 或 decision heading：直接走高顯著性機器契約通道，不用先嘗試低顯著性分流。
3. 碰完整報告正文或報告模板，且使用者會直接閱讀：走混合層報告呈現通道，補報告儲存、preview 或 mode template 測試。
4. 跨層改動、黃色採用訊號或紅色採用訊號：先填案例卡或拆分 patch，再決定是否跑多組命令。
5. 只有文件層或測試契約變更：可用文件契約測試驗證，但完成回報要說明尚未驗證 runtime 層與使用者行為層。

目的校準：

1. 降低 2 分鐘選通道摩擦：讓日常維護先照決策樹分流，而不是讀完整矩陣。
2. 保住高顯著性契約：只要碰 parser/prompt/template 或核心契約詞，仍直接升級。
3. 防止綠燈擴張：每個分流結果都要保留不得解讀為，不把單一測試通過寫成全面安全。
4. 保留低顯著性效率：純前端顯示層不被迫跑高顯著性全矩陣。

效用校準：

| 規則 | 預期效用 | 成本 | 升級或停用條件 |
|---|---|---|---|
| 一頁摘要優先 | 降低日常低風險改動的選通道成本。 | 可能讓操作者忽略跨層細節。 | 只要碰 parser/prompt/template、完整報告正文或兩層以上改動，就停用輕量路徑。 |
| 高顯著性機器契約通道 | 保護 parser、prompt、audit 與 conformance 契約。 | 測試較重，review 成本較高。 | 只有完全不碰機器契約詞與 template decision heading 時，才可降級。 |
| 混合層報告呈現通道 | 保護使用者會閱讀的正文、模板、儲存與 preview 流程。 | 需要同時看文案語氣與報告管線測試。 | 若同時碰 parser/prompt/template，升級到高顯著性或拆分 patch。 |
| 案例卡或拆分 patch | 降低跨層改動漏跑命令與限制句缺漏。 | 增加填寫與 review 成本。 | 純低顯著性顯示層且採用訊號為綠色時，可停用案例卡。 |
| 證據分層回報 | 防止綠燈擴張與工具化幻覺。 | 完成敘述較長。 | 不停用；若無 runtime 或使用者行為證據，就明確寫未驗證。 |

## 契約矩陣第 3 輪證據校準與觀測統計

信賴區間：

1. 目前樣本：本決策樹目前只由文件契約、HCS 狀態測試與前端顯示層回歸測試支撐。
2. 不可外推：不得外推成所有 review 都會正確選通道、所有 runtime 路徑已驗證、或所有使用者都會理解限制句。
3. 觀察窗口：每次契約相關變更都可以記錄一次分流結果；至少觀察 5 個案例後，才能討論是否調整決策樹。

相關性：

| 觀測訊號 | 可支持的判斷 | 不可推論 |
|---|---|---|
| 選通道時間下降 | 決策樹可能降低維護者第一次判斷成本。 | 不代表通道一定選對，也不代表測試範圍足夠。 |
| 錯選通道下降 | 分流規則可能改善改動層級判斷。 | 不代表因果；也可能是樣本較簡單或 reviewer 較熟。 |
| 限制句出現率上升 | 證據分層回報可能改善完成敘述品質。 | 不代表限制句內容正確，或 runtime 層已驗證。 |
| 案例卡觸發率上升 | 跨層改動可能更常被辨識出來。 | 不代表案例卡本身降低風險；仍需看漏跑命令與後續補救。 |

描述統計：

| 欄位 | 記錄方式 | 用途 |
|---|---|---|
| 樣本數 | 計算觀察窗口內契約相關變更案例數。 | 判斷目前統計是否只是零散個案。 |
| 中位選通道時間 | 記錄改動者從讀需求到選定通道的中位分鐘數。 | 避免單一極端慢案例扭曲 2 分鐘目標。 |
| 錯選率 | 錯選通道案例數 / 樣本數。 | 觀察決策樹是否降低分流錯誤。 |
| 跨層改動比例 | 跨兩層以上案例數 / 樣本數。 | 判斷案例卡或拆分 patch 的使用壓力。 |
| 案例卡觸發率 | 觸發案例卡案例數 / 跨層或黃色紅色訊號案例數。 | 檢查案例卡是否真的在需要時出現。 |
| 限制句出現率 | 完成回報包含不得解讀為的案例數 / 樣本數。 | 檢查證據分層回報是否被實際採用。 |

## 契約矩陣第 3 輪風險機率與顯著性門檻

機率：

| 風險 | 觀測欄位 | 風險機率判讀 |
|---|---|---|
| 錯選率 | 錯選通道案例數 / 樣本數。 | 若錯選率在至少 5 個案例後仍超過 20%，視為高風險機率，需回到決策樹檢查分流問題是否太模糊。 |
| 限制句缺漏率 | 未包含 `不得解讀為` 限制句的完成回報數 / 樣本數。 | 若限制句缺漏率超過 0%，先視為完成回報品質風險；若連續兩個觀察窗口仍缺漏，需升級 review 檢查。 |
| 案例卡漏觸發率 | 應觸發但未觸發案例卡的跨層或黃色紅色案例 / 跨層或黃色紅色案例。 | 若案例卡漏觸發率超過 10%，代表跨層風險可能被低估，不能只靠低顯著性顯示層通道放行。 |

迴歸：

1. 連續兩個觀察窗口都出現錯選率上升、限制句缺漏率上升或案例卡漏觸發率上升，才視為穩定回歸訊號。
2. 回歸監測必須同時看跨層改動比例；若樣本從純前端顯示層轉成混合層或高顯著性機器契約改動，不能把錯選率上升直接歸因於決策樹變差。
3. 若單一窗口出現紅色高風險案例，例如 parser/prompt/template 改動被放入低顯著性通道，可立即升級，不必等待第二個窗口。

顯著性：

1. 小樣本：少於至少 5 個案例時不得宣稱改善，只能描述個案與待觀察風險。
2. 升級門檻：至少 5 個案例後，若錯選率超過 20%、限制句缺漏率大於 0%、或案例卡漏觸發率超過 10%，需升級到人工 review 檢查。
3. 調整決策樹：只有在連續兩個觀察窗口都有同方向回歸，且已排除樣本層級改變時，才調整決策樹文字或案例卡觸發規則。
4. 不得宣稱改善：測試通過、文件章節存在或單一案例變好，不得宣稱改善分流品質、使用者理解或 runtime 安全。

## 契約矩陣第 3 輪證據規則與外推邊界

證據基礎：

| 證據類型 | 可接受證據 | 可支持的判斷 |
|---|---|---|
| 文件契約測試 | `tests/test_docs_contract.py` 與 `tests/test_hcs_plus_state.py` 通過，且章節包含門檻、限制句與下一步 checkpoint。 | 文件契約未被移除，HCS 輪巡狀態可追蹤。 |
| 觀察窗口紀錄 | 每個契約相關變更都記錄樣本數、改動層級、錯選率、限制句缺漏率與案例卡漏觸發率。 | 可討論分流品質是否需要人工 review 或後續調整。 |
| 案例卡 | 跨層、黃色或紅色案例明確列出改動描述、選擇通道、必跑命令、限制句與採用訊號。 | 可支持單一案例是否需要升級、拆分 patch 或補跑測試。 |

不可作為證據：

1. 單次綠燈不能作為分流品質改善、真實使用者理解或 runtime 安全的證據。
2. 未標樣本數的比例不能作為錯選率下降、限制句品質改善或案例卡採用改善的證據。
3. 只看文件章節存在，不能作為 reviewer 已採用、生成報告已安全、或所有 pipeline 模式都被驗證的證據。

演繹：

1. 若碰 parser/prompt/template、`[投資建議]`、`[/投資建議]` 或 decision heading，立即升級到高顯著性機器契約通道，不能等待兩個觀察窗口。
2. 若少於至少 5 個案例，只能描述個案、列出待觀察風險與保留限制句，不能演繹出決策樹已改善。
3. 若連續兩個觀察窗口同方向回歸，且跨層改動比例沒有同向升高，才可演繹為決策樹或案例卡規則需要調整。

歸納：

1. 外推邊界：文件契約測試只能歸納到目前文件與 HCS 狀態仍保留護欄，不得外推到 runtime 安全。
2. 觀察窗口紀錄只能歸納到已記錄案例，不得外推到未記錄改動、真實使用者理解或全部 review 行為。
3. 案例卡只能歸納到代表性跨層案例，不得外推到生成報告母體、歷史輸出或未來 LLM 回覆。

## 契約矩陣第 3 輪反謬誤與來源情境邊界

謬誤：

| 易犯謬誤 | 錯誤推論 | 反謬誤護欄 |
|---|---|---|
| 測試綠燈謬誤 | 文件契約測試通過，所以分流品質、runtime 安全或真實使用者理解都已改善。 | 測試綠燈只能證明文件護欄仍存在；完成回報限制句必須寫明不得替代 runtime 驗證與不得替代使用者研究。 |
| 樣本數謬誤 | 錯選率、限制句缺漏率或案例卡漏觸發率看起來下降，所以決策樹已改善。 | 未標樣本數、改動層級與觀察窗口的比例不得作為完成證據；少於至少 5 個案例只能描述個案。 |
| 案例代表性謬誤 | 一張案例卡處理正確，所以所有跨層改動或生成報告母體都安全。 | 案例卡只支持該代表性案例的升級、拆分或補測判斷，不得外推到未記錄案例。 |

來源品質：

| 來源等級 | 可使用來源 | 不得作為完成證據 |
|---|---|---|
| 高品質來源 | 可重跑測試輸出、契約相關 diff、包含樣本數與改動層級的觀察窗口紀錄、完整案例卡。 | 只看測試名稱、只看章節存在、口頭說明已 review。 |
| 次級來源 | 人工 review 註記、操作者回饋、單一案例觀察、未自動化的文件審閱。 | 次級來源不得單獨支持「流程改善」或「使用者理解改善」。 |
| 不得作為完成證據 | 單次綠燈、未標樣本數比例、未列改動層級的觀察、未寫限制句的完成回報。 | 不得用來支持合併、降級通道、停用案例卡或宣稱安全。 |

情境脈絡：

1. 只適用於契約相關變更：parser/prompt/template、報告正文、跨層改動、案例卡或分流決策變更。
2. 不適用於一般 UI 文案：純排版、靜態樣式或不碰報告語意的顯示微調可用一般前端測試與 review，不必填完整契約矩陣。
3. 需要人工 review：任何黃色或紅色採用訊號、跨層改動、核心契約詞改動或樣本數不足卻要調整規則的情境，都要人工 review。
4. 不得替代 runtime 驗證：文件契約測試與案例卡不能證明報告生成、parser 實際輸出或前端 runtime 行為安全。
5. 不得替代使用者研究：觀察窗口與操作者回饋不能直接證明真實使用者理解，也不能證明使用者不會把報告當成交易指令。

## 契約矩陣第 3 輪負擔估算與完成詮釋框架

批判：

| 矩陣負擔 | 處理方式 | 判斷 |
|---|---|---|
| 必留護欄 | parser/prompt/template、核心契約詞、跨層改動、黃色或紅色採用訊號仍需完整通道、案例卡或人工 review。 | 這些情境的錯放成本高於填寫成本。 |
| 可短句替代 | 低風險 UI、純排版、靜態樣式或不碰報告語意的顯示微調，只需一句「不碰 parser/prompt/template，走前端測試」。 | 避免矩陣過重拖慢一般顯示層改動。 |
| 可延後工具化 | 自動選測、案例卡資料庫、觀察窗口儀表板先不做。 | 目前樣本仍少，先保留文件契約與人工 review，比新增工具更合理。 |

估算：

| 情境 | 完成回報成本 | 建議上限 | 最小證據 |
|---|---|---|---|
| 低風險 UI | 1 句通道判斷 + 前端測試結果。 | 2 分鐘 | `tests/test_static_history_filters.py` 或相關前端測試。 |
| 混合層報告呈現 | 2 到 3 句：改動層級、命令、不得宣稱安全限制句。 | 3 分鐘 | 報告模板、儲存或 preview 相關測試。 |
| 高風險契約 | 4 句以上：改動層級、必跑命令、案例卡或人工 review、不得替代 runtime/使用者研究限制句。 | 不設硬上限 | parser/prompt/template diff、完整測試輸出與必要案例卡。 |

詮釋框架：

| 證據狀態 | 完成回報詮釋 | 禁止宣稱 |

| runtime 驗證通過 | 指定 runtime 路徑在本次命令中通過。 | 不得推論全部生成報告母體安全。 |
| 使用者研究完成 | 可描述該研究樣本中的理解結果。 | 不得替代 parser/prompt/template 契約驗證。 |

## 契約矩陣第 3 輪創意學習入口

限制條件：

1. 不改 runtime 行為：本入口只改文件契約與學習順序，不改 parser、prompt、template、報告生成或前端互動。
2. 不新增自動選測腳本：仍由操作者判斷改動層級，再複製對應命令。
3. 不新增遙測：5 分鐘復盤先以人工觀察窗口與案例卡記錄，不新增產品遙測或背景收集。

5. 不把文件契約通過解讀成 runtime 安全、使用者理解改善或 HCS Plus 完成。

類比：

契約矩陣像登機前安檢：它先把人流分到快速通道或人工複檢，要求把必要證據放進證據托盤，但不把安檢通過解讀成航程安全。

| 登機前安檢類比 | 系統對應 | 不可外推 |
|---|---|---|
| 快速通道 | 低風險 UI、純排版、靜態樣式或不碰報告語意的顯示微調。 | 快速通道通過不代表 parser/prompt/template 安全。 |
| 人工複檢 | 黃色/紅色採用訊號、跨層改動、核心契約詞或高風險契約。 | 人工複檢不能省略測試命令、案例卡或限制句。 |
| 證據托盤 | 測試輸出、diff、案例卡、觀察窗口與完成回報限制句。 | 證據放齊只代表本次檢查可追溯，不把安檢通過解讀成航程安全。 |

## 契約矩陣第 3 輪操作演算法與捷思規則

演算法：

| 四步操作演算法 | 操作 | 產出 |
|---|---|---|
| 步驟 1：10 秒判斷 | 先問是否碰 parser/prompt/template、核心契約詞、跨層改動、黃色或紅色訊號。 | 標成快速通道、人工複檢或停止補證據。 |
| 步驟 2：選擇通道 | 快速通道走前端/顯示層測試；人工複檢走高風險契約或混合層測試；停止補證據先拆 patch 或補案例卡。 | 一個通道名稱與一組最小命令。 |
| 步驟 3：裝好證據托盤 | 放入測試輸出、diff、案例卡、觀察窗口與完成回報限制句；缺哪個就標明為待補證據。 | 可讓 reviewer 追溯的證據包。 |
| 步驟 4：完成回報 | 用通道、命令、結果、限制句收尾；缺少限制句就不得完成。 | 完成回報不宣稱安全、不宣稱理解改善、不得替代 runtime 驗證。 |

設計思考：

| 三個操作者情境 | 預設路徑 | 回報焦點 |
|---|---|---|
| 情境 A：只改低風險 UI | 只在前端顯示才走快速通道，執行前端顯示層測試。 | 說明不碰 parser/prompt/template，列出前端測試結果。 |
| 情境 B：改報告模板或正文呈現 | 走混合層報告呈現通道，檢查模板、儲存或 preview 相關測試。 | 說明報告呈現路徑通過，但不得替代 runtime 驗證。 |
| 情境 C：改 parser、prompt、template 或核心契約詞 | 有核心契約詞就先人工複檢，優先補高風險契約證據。 | 列出 parser/prompt/template diff、必跑命令、案例卡與限制句。 |

捷思法：

三條快速規則：

1. 有核心契約詞就先人工複檢：碰 `[投資建議]`、prompt、parser regex、template decision heading 或核心契約詞時，不走快速通道。
2. 只在前端顯示才走快速通道：只有純排版、靜態樣式、低風險 UI 或不碰報告語意的顯示微調，才可用短句與前端測試收尾。
3. 缺少限制句就不得完成：完成回報若沒有不得宣稱安全、不得宣稱理解改善或不得替代 runtime 驗證，就回到步驟 3 補證據托盤。

## 契約矩陣第 3 輪採用最佳化與訊號板

最佳化：

| 採用摩擦 | 觀察問題 | 最小調整 |

| 錯選通道 | 操作者是否把高風險契約改動放進快速通道，或把低風險 UI 過度升級？ | 調整四步操作演算法的第一步提示。 |

## 契約矩陣第 3 輪案例模型與抽樣案例卡

建模：

| 代表性案例模型 | 觸發條件 | 必看證據 |
|---|---|---|
| 模型 A：低風險快速通道案例 | 只改低風險 UI、純排版、靜態樣式或不碰報告語意的顯示微調。 | 前端顯示層測試、短句回報、不碰 parser/prompt/template。 |
| 模型 B：混合層報告呈現案例 | 改報告模板、Markdown/HTML 正文呈現、preview 或儲存路徑。 | 混合層命令、報告呈現 diff、不得替代 runtime 驗證限制句。 |
| 模型 C：高風險契約人工複檢案例 | 改 parser、prompt、template、核心契約詞或跨層契約。 | parser/prompt/template diff、必跑命令、案例卡、人工 review 與限制句。 |
| 模型 D：紅色阻擋案例 | 高風險契約被放進快速通道、核心契約詞缺人工複檢，或完成回報宣稱安全/理解改善。 | 停止合併原因、補救行動、重新選通道與不可外推範圍。 |

抽樣：

1. 代表性抽樣規則：每個觀察窗口至少保留一個實際出現的低風險快速通道案例、混合層報告呈現案例與高風險契約人工複檢案例。
2. 黃色或紅色必抽：任何黃色或紅色採用訊號、跨層改動、核心契約詞改動或案例卡漏補，都要抽成案例卡。
3. 若某模型在該觀察窗口未出現，記錄「本窗口未觀察到」，不得用其他模型替代。
4. 少於 5 個案例不得宣稱趨勢；只能描述個案與待觀察風險。

個案研究：

| 案例卡格式 | 必填內容 |
|---|---|
| 改動描述 | 這次改了哪個文件、測試、前端、模板、parser、prompt 或契約。 |
| 改動層級 | 低風險 UI、混合層報告呈現、高風險契約人工複檢或紅色阻擋。 |
| 選擇通道 | 實際走快速通道、混合層通道、人工複檢或停止合併。 |
| 證據托盤 | 測試輸出、diff、案例卡、人工 review、觀察窗口或必要限制句。 |
| 採用訊號 | 綠色、黃色或紅色，以及觸發原因。 |
| 限制句 | 不得宣稱 runtime 安全、使用者理解改善、測試覆蓋完整或 HCS Plus 完成。 |
| 補救行動 | 補案例、補提示、重寫步驟、重跑命令、拆分 patch 或回到人工複檢。 |
| 不可外推 | 本案例不得外推到未觀察窗口、未記錄 review、生成報告母體或產品遙測。 |

| 報告呈現維護者 | 混合層報告呈現案例、實際執行命令、不可外推。 | 報告呈現變動不能只用前端顯示測試替代。 |
| 契約複檢維護者 | 高風險契約人工複檢案例、人工複檢、停止合併條件。 | 入口不是自動判斷器；核心契約詞仍需人工 review。 |
| 觀察流程維護者 | 觀察記錄欄位、複製檢查清單、可複製完成條件。 | 觀察欄位不是 pytest；複製成功不是改善證明。 |

組成：

1. 第一步：先判斷讀者角色，確認自己是在低風險 UI、報告呈現、契約複檢或觀察流程維護情境。
2. 第二步：只讀對應入口，避免低風險 UI 維護者被完整矩陣拖慢，也避免高風險契約被誤放快速通道。
3. 第三步：補齊觀察欄位，至少保留選定案例模型、實際選擇通道、實際執行命令與完成回報三欄。
4. 第四步：用限制句收尾，說清楚不得宣稱 runtime 安全、使用者理解改善、流程改善或 HCS Plus 完成。

語意含義：

| 詞句 | 正確含義 | 不得誤讀為 |
|---|---|---|
| 讀者角色不是權限等級 | 角色只用來縮短閱讀路徑。 | 不是誰可以跳過證據，也不是誰可以覆蓋紅色訊號。 |
| 入口不是自動判斷器 | 入口幫助維護者先定位情境。 | 不是工具化選測，也不是自動分類結果。 |
| 觀察欄位不是 pytest | 觀察欄位保存人工 review 脈絡。 | 不能替代測試、lint、build 或人工 review。 |
| 複製成功不是改善證明 | 複製成功只代表下一位操作者能用同一欄位重做紀錄。 | 不代表錯選通道下降、使用者理解改善或 runtime 安全。 |
| 低風險不代表低責任 | 低風險 UI 可用短句回報，但仍要保留限制句。 | 不是可以省略不碰 parser/prompt/template 的說明。 |

本讀者語意入口只改善閱讀路徑與語意邊界，不新增產品遙測，不替代 pytest 或人工 review，不得宣稱 runtime 安全、不得宣稱使用者理解改善，也不得宣稱 HCS Plus 完成。

## 契約矩陣第 3 輪維護導覽與核心論點

組織結構：

章節導覽：

1. 先定位讀者角色：從低風險 UI 維護者、報告呈現維護者、契約複檢維護者、觀察流程維護者四種入口選一個。
2. 再選通道與案例模型：對照快速通道、混合層報告呈現、高風險契約人工複檢或紅色阻擋案例。
3. 接著補觀察欄位：至少保留選定案例模型、實際選擇通道、實際執行命令與完成回報三欄。
4. 最後用限制句與核心論點收尾：說清楚不得宣稱改善、runtime 安全、使用者理解改善或 HCS Plus 完成。

專業性：

維護語氣：

| 語氣要求 | 建議寫法 | 避免寫法 |
|---|---|---|
| 只描述觀察窗口 | 本觀察窗口記錄了哪些案例、通道、命令與限制句。 | 本流程已證明所有未來改動都會改善。 |
| 明列未跑命令 | 未跑的命令要列出原因，並說明該證據層仍未驗證。 | 只寫測試通過，省略未涵蓋的契約面。 |
| 把紅色訊號說成停止條件 | 紅色訊號代表停止合併、回到人工複檢或補跑命令。 | 把紅色訊號寫成可後補的低風險提醒。 |

論點：

核心主張：

1. 契約矩陣的目的不是提高文件厚度，而是讓低風險改動更快收尾、讓高風險契約更早升級、讓觀察紀錄可複製但不被誤讀。
2. 低風險 UI 可以用短句與最小命令完成，但必須保留不碰 parser/prompt/template 的限制句。
3. 高風險契約一旦碰到 parser、prompt、template 或核心契約詞，就要回到人工複檢與對應 pytest，不得用觀察紀錄替代。
4. 本章節只提供維護導覽與回報語氣，不得宣稱改善，不得替代 pytest 或人工 review。

表達：

建議句型：

## 契約矩陣第 3 輪倫理阻擋與責任判斷

倫理考量：

短版回報倫理底線：

1. 不得把短版回報寫成安全背書；短版只能說明本次改動層級、已跑命令、未跑命令與不得解讀為。
2. 不得把責任轉嫁給文件、工具或測試；改動者仍要說明通道選擇、證據層與限制句。
3. 不得用快速通道淡化高風險契約；碰到 parser、prompt、template 或核心契約詞時，快速通道不能成立。

倫理勇氣：

必要時要說不：

1. 缺少 parser/prompt/template 證據時停止合併，不用短版回報包裝成已完成。
2. 報告文案像交易指令時先補責任邊界，再判斷是否能走低風險或混合層通道。
3. 高風險契約被降級時回到人工複檢，必要時拆分 patch 或補跑高顯著性命令。

倫理判斷：

| 判斷項 | 允許回報 | 禁止回報 |
|---|---|---|
| 短版回報 | 已列出通道、已跑命令、未跑命令與不得解讀為。 | 短版回報已填，所以系統安全或使用者已理解。 |
| 快速通道 | 本次只碰前端顯示層，未碰 parser、prompt、template 或核心契約詞。 | 快速通道代表可以省略責任邊界或人工判斷。 |
| 文件與觀察 | 文件或觀察保存了 review 脈絡，仍需對應 pytest 或人工 review。 | 文件或觀察足以替代測試、lint、build 或人工 review。 |

升級判斷：

1. 低風險改動若碰到使用者行動暗示，升級到混合層報告呈現，並補責任邊界。
2. 混合層若碰到核心契約詞，升級到高風險契約人工複檢，並補跑對應 pytest。
3. 文件或觀察若被拿來宣稱 runtime 行為，必須升級為 runtime 驗證或人工驗收；不得宣稱使用者已理解，不得替代 pytest 或人工 review。

## 契約矩陣第 3 輪系統因果與證據層次

複雜因果：

局部綠燈因果圖：

| 局部訊號 | 可能造成的錯誤推論 | 必要修正 |
|---|---|---|
| 文件契約通過 | 文件契約通過可能造成流程已安全的錯誤推論。 | 只能說文件章節與狀態紀錄存在；若要宣稱流程被使用，需實際 review 或案例紀錄。 |
| 前端測試通過 | 前端測試通過可能造成 parser/prompt 已安全的錯誤推論。 | 只支持前端顯示層契約；碰 parser、prompt、template 時必須升級測試。 |
| 倫理阻擋存在 | 倫理阻擋存在可能造成高風險已被完全阻擋的錯誤推論。 | 只能說阻擋條件已寫入契約；仍需 reviewer 啟用與補證據行動。 |

湧現特性：

1. 低風險快速通道累積成高風險語氣漂移：單次文案可走快速通道，多次累積後可能改變整體責任感。
2. 案例卡增加但實際驗證減少：文件與觀察越完整，操作者越可能誤以為不必重跑 pytest 或人工 review。
3. 阻擋規則存在但 reviewer 不敢啟用：規則寫得越完整，若權責不清，仍可能在壓力下被跳過。

分析層次：

| 層次 | 可支持的宣稱 | 不可替代的層次 |
|---|---|---|
| 文件層 | 章節、限制句、案例卡與狀態紀錄存在。 | 不得用文件完整替代 runtime 驗證。 |
| 測試層 | 指定 pytest、lint 或靜態契約未被已知案例打破。 | 不得用測試通過宣稱使用者理解。 |
| runtime 層 | 實際執行路徑、生成報告或 UI 呈現已被驗證。 | 不得替代使用者行為層的理解與採用證據。 |
| 使用者行為層 | 使用者或 reviewer 如何理解、採用、阻擋或升級。 | 不得反向替代 parser、prompt、template 或 runtime 測試。 |

層次規則：

1. 同層證據只能支持同層宣稱；文件層只支持文件存在，測試層只支持指定測試範圍。
2. 跨層宣稱必須升級驗證；若要從文件層宣稱 runtime 行為，必須補 runtime 測試或人工驗收。
3. 任何完成回報若跨層外推，都要回到倫理阻擋與責任判斷，改寫為同層宣稱或補證據。

## 契約矩陣第 3 輪維護網絡與動態圖像

網絡：

維護網絡：

| 節點 | 連到哪裡 | 主要風險 |
|---|---|---|
| 文件層節點 | 連到測試層節點與 reviewer 阻擋節點。 | 文件完整被誤讀為流程安全。 |
| 測試層節點 | 連到 runtime 層節點與跨層宣稱升級回路。 | 測試通過被誤讀為使用者理解或 parser/prompt 安全。 |
| runtime 層節點 | 連到使用者行為層節點與人工驗收。 | 實際執行成功被誤讀為使用者一定正確採用。 |
| 使用者行為層節點 | 連到 reviewer 阻擋節點與責任邊界。 | 使用者或 reviewer 的理解被口頭推測取代。 |
| reviewer 阻擋節點 | 連到倫理阻擋、人工複檢與補證據行動。 | 阻擋規則存在但壓力下未被啟用。 |

系統動力學：

1. 快速通道摩擦降低回路：短版回報降低低風險 UI 摩擦，但若累積語氣漂移，必須回到混合層或高風險檢查。
2. 案例卡形式化回路：案例卡增加可追溯性，但若實際驗證減少，必須補跑 pytest 或人工 review。
3. 阻擋勇氣回路：倫理阻擋讓 reviewer 有理由說不，但若說不成本過高，阻擋節點會失效。
4. 跨層宣稱升級回路：一旦文件層或測試層想宣稱 runtime 或使用者行為，就必須升級驗證。

系統圖像：

1. 先定位證據層：文件層、測試層、runtime 層或使用者行為層。
2. 再連到網絡節點：找出牽動文件、測試、runtime、使用者行為或 reviewer 阻擋的哪個節點。
3. 接著判斷動態回路：檢查快速通道、案例卡、阻擋勇氣或跨層宣稱哪個回路正在作用。
4. 最後決定維持同層宣稱或升級驗證：能同層說清楚就保留短版回報；跨層宣稱就補測試、人工驗收或使用者行為證據。

本網絡圖像只輔助維護判斷，不得把網絡圖像當成自動審核器，不得替代 pytest 或人工 review。

## 契約矩陣第 3 輪 review 對話與預設行為

談判：

補證據協商不是降低標準；它要求不降低標準，而是把「不能合併」改成可接受的下一步。reviewer 可以接受同層宣稱、限制句或拆分 patch，但不能用好聽句型替代證據。

| 情境 | 可說句型 | 不可說法 |
|---|---|---|
| 文件或測試想跨層宣稱 | 我可以接受同層宣稱，但跨層宣稱需要補證據。 | 文件已補完整，所以流程安全。 |
| 快速通道想保留低摩擦 | 若要保留快速通道，請移除 runtime 或使用者行為宣稱。 | 只是文案，不需要限制句。 |
| parser/prompt 或高顯著性安全宣稱 | 若要宣稱 parser/prompt 安全，請補高顯著性命令或拆分改動。 | 前端測試過了，所以 parser/prompt 安全。 |

說服：

說服不是美化風險，而是降低說不成本，讓補證據看起來像共同完成工作，而不是否定改動者。

說服路徑：

1. 先承認已完成的證據：哪些文件、測試、案例卡或限制句已存在。
2. 再指出缺口：哪個宣稱越過文件層、測試層、runtime 層或使用者行為層。
3. 接著提出最小可接受補證據：補跑命令、補限制句、拆分 patch、降級宣稱或人工 review。
4. 最後寫不得解讀為：保留同層結論，但明示不能外推到 runtime、parser/prompt、使用者理解或投資判斷安全。

形塑行為：

預設行為：

| 訊號 | 預設動作 | 完成條件 |
|---|---|---|
| 綠色 | 維持同層宣稱。 | 完成回報預設三欄：本次宣稱層級、已補證據、仍不得解讀為。 |
| 黃色 | 黃色：同層宣稱可合併但補限制句。 | 限制句明確指出未驗證的層次。 |
| 紅色 | 紅色：停止合併、補跑 pytest 或人工 review、拆分 patch。 | 補證據後重新定位證據層與網絡節點。 |
| 跨層宣稱 | 跨層宣稱預設升級。 | 補 runtime 驗證、人工驗收、使用者行為證據或降級成同層宣稱。 |

本章只提供 review 對話、說服路徑與預設行為；不得把好聽句型當成證據，不得替代 pytest 或人工 review。

## 契約矩陣第 3 輪防從眾、差異訊號與情緒調節

從眾：

防從眾檢查：

| 壓力訊號 | 不可取代 | 必要回應 |
|---|---|---|
| 多數同意 | 多數同意不是證據。 | 回到本次宣稱層級、已補證據與仍不得解讀為。 |
| 前例綠燈 | 前例綠燈不是本次綠燈。 | 重新定位本次改動層級、證據層與網絡節點。 |
| 測試全綠 | 測試全綠不是限制句。 | 保留 pytest 結果，同時補上不得外推的限制句。 |
| 合併壓力 | 快要合併不是降低標準的理由。 | 降級成同層宣稱、補最小證據或拆分 patch。 |

差異：

差異訊號：

1. 改動層級差異：高顯著性、混合層、低顯著性不得合併成同一種風險。
2. 證據層差異：文件層、測試層、runtime 層與使用者行為層要分開回報。
3. pipeline 模式差異：長線、交易、逆勢、短線模式要保留不同責任邊界。
4. 風險顏色差異：綠色、黃色、紅色訊號要分別處理，不得把黃色與紅色訊號寫成綠色。

情緒智商：

高壓語氣處理：

1. 先命名壓力來源：時程壓力、回歸失敗、review 疲勞、權威催促或合併窗口。
2. 再回到預設三欄：本次宣稱層級、已補證據、仍不得解讀為。
3. 接著保留最小補證據路徑：補限制句、補跑命令、人工 review、降級宣稱或拆分 patch。
4. 最後用冷靜限制句收尾：目前證據支持同層宣稱，但不得外推到未驗證層次。

不得用趕時間取代證據層，不得用情緒安撫取代 pytest 或人工 review。

## 契約矩陣第 3 輪角色責任與權力護欄

領導原則：

證據領導：

| 角色 | 必須主導的動作 | 不得替代的事 |
|---|---|---|
| 主責 | 主責先宣告本次宣稱層級，並列出已補證據。 | 不以「這只是小改」取代證據層定位。 |
| review 主導者 | review 主導者維持升級權，遇到跨層宣稱、黃色或紅色訊號時要求升級或拆分。 | 不以「大家都同意」取代補證據。 |
| 合併者 | 合併者確認紅色與黃色訊號已處理，並確認仍不得解讀為存在。 | 不以速度領導取代證據領導。 |

權力動態：

1. 合併權限不能覆蓋紅色訊號；紅色訊號必須先停止合併、補證據或拆分 patch。
2. 資深度不能把前例綠燈變成通行證；每次仍要回到本次證據層與改動層級。
3. 低權限操作者可以引用契約要求補證據；這不是阻礙，而是維持系統證據邊界。
4. 權威催促必須回到預設三欄：本次宣稱層級、已補證據、仍不得解讀為。

責任：

| 角色 | 責任 | 完成回報必留 |
|---|---|---|
| 改動者 | 改動者負責本次宣稱層級與已補證據。 | 本次改動層級、已跑命令或未跑命令。 |
| reviewer | reviewer 負責仍不得解讀為，並核對黃色、紅色與跨層宣稱。 | 不得外推到 runtime、parser/prompt、使用者理解或投資判斷安全。 |
| 合併者 | 合併者負責未跑命令與剩餘風險，並確認是否可接受。 | 剩餘風險、延後理由與後續補證據入口。 |

問題可追溯到角色責任，不得把責任轉嫁給文件、工具或測試，不得替代 pytest 或人工 review。

## 契約矩陣綜合優化第 1 次：驗證、溝通與系統圖像收斂

綜合視角：

- #可驗證性：所有完成宣稱都要回到同層證據、已跑命令或明確限制句。
- #溝通設計：完成回報先讓維護者知道本次能宣稱什麼，再說不能外推什麼。

最終操作收斂：

驗證閘門：

| 改動層 | 必留證據 | 不得宣稱 |
|---|---|---|
| 低風險同層改動 | 完成回報三欄、文件或靜態測試。 | 不得宣稱 parser、prompt、template 或 runtime 安全。 |

1. 本次宣稱層級：說明改動屬於前端顯示層、報告呈現層、機器契約層或維運決策層。
2. 已補證據：列出已跑命令、已補文件、已做人工 review 或明確未跑命令。
3. 仍不得解讀為：寫出不得外推到 runtime、parser/prompt/template、使用者理解或投資判斷安全。

系統圖像收斂：

|---|---|---|
| 前端顯示層 | 靜態文案、tab、panel、低風險 UI wiring。 | 靜態 DOM/JS/CSS 測試與完成回報三欄。 |
| 報告呈現層 | preview、history、compare、rerun、report label。 | 報告呈現測試、文件契約與限制句。 |

驗收標準：

1. 每個完成宣稱都有對應命令或限制句。
2. 高顯著性機器契約改動仍跑 parser、prompt、template 與 audit 回歸。
3. 低風險同層改動保持輕量三欄，不被完整矩陣拖慢。
4. 不得把綜合優化第 1 次解讀為 HCS Plus 完成。

## 契約矩陣綜合優化第 2 次：證據來源、讀者角色與責任承接

綜合視角：

- #證據基礎：先分清直接證據、間接證據、缺口證據與未跑命令，避免把文件完整誤當 runtime 安全。
- #受眾：不同維護者需要不同入口，低風險 UI 維護者不該讀完整機器契約，機器契約維護者不能只看短版回報。
- #責任：改動者、reviewer、合併者要分別承接證據來源、讀者誤讀與剩餘風險。

證據來源分級：

| 證據類型 | 可支持的宣稱 | 不得外推 |
|---|---|---|
| 直接證據 | 對應測試命令、人工 review 或實際檢查覆蓋的同層行為。 | 不得外推到未測層、使用者理解、安全或投資判斷。 |
| 間接證據 | 文件契約、案例卡、觀察紀錄或靜態檢查支持流程存在。 | 不得宣稱 runtime、parser/prompt/template 或使用者行為已安全。 |
| 缺口證據 | 已知道缺少的命令、未做的人工 review 或未驗證層次。 | 不得把缺口寫成綠燈。 |
| 未跑命令 | 完成回報必列未跑命令與延後理由。 | 未跑命令不能消失，也不能被「測試全綠」概括掉。 |

讀者角色分流：

| 讀者角色 | 先讀內容 | 必須保留 |
|---|---|---|
| 低風險 UI 維護者 | 輕量三欄與前端顯示層證據。 | 不得把 UI 綠燈外推到 parser、prompt、template 或 runtime。 |
| 報告呈現維護者 | 報告呈現層、限制句與使用者誤讀風險。 | 不得把報告文案寫成交易指令或安全背書。 |
| 機器契約維護者 | parser、prompt、template、audit 與核心契約詞。 | 高顯著性命令與人工 review 不能省略。 |
| 維運決策維護者 | watchlist、daily decision、active jobs、operator summary 與剩餘風險。 | 未跑命令與排程風險必須留到下一步。 |
| 合併者 | 證據來源、讀者角色、未跑命令與剩餘風險。 | 只能接受明確責任，不能接受模糊安全宣稱。 |

責任承接：

1. 改動者負責證據來源與宣稱層級：說明哪些是直接證據、間接證據、缺口證據與未跑命令。
2. reviewer 負責讀者是否會誤讀：檢查低風險 UI 維護者、報告呈現維護者、機器契約維護者與維運決策維護者是否會讀錯層級。
3. 合併者負責未跑命令與剩餘風險是否可接受：未跑命令不能消失，剩餘風險必須留到下一步。
4. 不得把使用者理解、安全或投資判斷外推，也不得把綜合優化第 2 次解讀為 HCS Plus 完成。
5. 本章不得新增 runtime、遙測或自動選測工具；它只收斂證據、讀者與責任邊界。

3. 未跑命令留到下一步：未跑命令要保留命令名稱、延後理由與限制的宣稱範圍。

## 契約矩陣綜合優化第 4 次：目標校準、效用門檻與合理性審核

綜合視角：

- #目的：契約矩陣必須服務股票研究系統核心目標，而不是服務文件本身。
- #效用：每條新規則都要能降低錯選模式、漏跑命令、跨層外推或維護成本。
- #合理性：高成本規則必須有證據，目的不明或低效用規則必須刪減。

目標校準：

| 目標 | 矩陣應服務的行為 | 不應服務 |
|---|---|---|
| 股票研究系統核心目標 | 讓操作者更快選對分析模式、理解報告層級與保留風險。 | 不為了增加文件厚度而新增規則。 |
| 使用者決策用途 | 協助判斷研究、交易、逆勢或短線事件的不同用途。 | 不把報告文字寫成交易指令或安全背書。 |
| 維護者合併判斷 | 幫助 reviewer 和合併者判斷證據、未跑命令與剩餘風險是否足夠。 | 不讓表格填寫取代實際測試、人工 review 或限制句。 |
| 契約安全邊界 | 保住 parser、prompt、template、audit 與核心契約詞的高顯著性邊界。 | 不用低風險入口處理高顯著性機器契約。 |

效用門檻：

| 門檻 | 必須能回答 | 低效用訊號 |
|---|---|---|

| 降低維護成本 | 這條規則是否縮短分流、刪掉不必要步驟或保留輕量通道？ | 只新增審核負擔，沒有降低風險。 |

2. 比例性：規則成本是否符合風險層級；低風險同層改動不得被高成本流程拖慢。
3. 可驗證性：規則是否能被文件契約測試、pytest、人工 review 或明確限制句驗證。
4. 可逆性：若規則造成流程膨脹，是否能刪除而不破壞核心契約。
5. 低效用規則必須刪減，高成本規則必須有證據，目的不明不能加入矩陣。

限制句：

- 不得讓契約矩陣服務文件本身。
- 不得把效用推論寫成已證明改善。
- 不得把綜合優化第 4 次解讀為 HCS Plus 完成。
- 不得新增 runtime、遙測或自動選測工具；本章只校準目的、效用與合理性。

## 契約矩陣綜合優化第 5 次：限制邊界、分流決策與成本最佳化

綜合視角：

- #限制條件：把前面幾批的不得解讀為、不得新增工具與不得替代驗證收斂成硬限制、軟限制、升級限制與停用限制。
- #決策樹：把目標校準、效用門檻與合理性審核排成可執行分流，而不是讓 reviewer 自行記憶整份矩陣。
- #最佳化：保留低風險輕量通道，合併重複規則，刪除低效用規則，延後無證據規則。

限制邊界：

| 限制類型 | 適用情境 | 必須動作 |
|---|---|---|
| 硬限制 | 任何修改企圖新增 runtime、遙測或自動選測工具，或把文件契約當成自動審核器。 | 不得新增 runtime、遙測或自動選測工具；不得替代 pytest 或人工 review。 |
| 軟限制 | 純前端顯示層、文件同層或低風險文案改動。 | 可走輕量通道，但必須保留宣稱層級、已補證據與限制句。 |
| 升級限制 | 觸及 parser、prompt、template、audit、核心契約詞、交易語氣或跨層宣稱。 | 必須升級到對應命令、人工 review 或拆分 patch；不得生成交易指令或安全背書。 |
| 停用限制 | 新規則目的不明、效用不足、證據缺口過大或成本高於風險。 | 停用或延後，不加入契約矩陣。 |

分流決策：

1. 第一步：判斷改動層級，是前端顯示層、報告呈現層、機器契約層或維運決策層。
2. 第二步：判斷顯著性，確認是否碰到 parser、prompt、template、audit、交易語氣或跨層宣稱。
3. 第三步：判斷證據缺口，分出直接證據、間接證據、缺口證據與未跑命令。
4. 第四步：選擇輕量、升級、拆分或刪減；低風險可輕量，高顯著性要升級，混合層先拆分，低效用規則刪減。

成本最佳化：

| 最佳化動作 | 觸發條件 | 保留底線 |
|---|---|---|
| 保留低風險輕量通道 | 改動只碰同層文案、文件或前端顯示，且沒有跨層宣稱。 | 仍需限制句與未跑命令。 |
| 合併重複規則 | 兩條規則服務同一目標、同一效用門檻與同一證據入口。 | 不得合併到看不出責任承接。 |
| 刪除低效用規則 | 規則不能降低錯選模式、漏跑命令、跨層外推或維護成本。 | 刪除前保留歷史決策理由。 |
| 延後無證據規則 | 規則看似有用，但沒有直接或間接證據支持。 | 留到下一步，不寫成已完成。 |

限制句：

- 不得為了最佳化而降低高顯著性驗證。
- 不得把決策樹當成自動選測工具。
- 不得把綜合優化第 5 次解讀為 HCS Plus 完成。

1. 文件與測試契約優先：後續新增 pipeline、報告模板、模式語意或完成回報規則時，先更新契約章節與對應測試。
2. 例外升級：碰到 parser、prompt、template、audit、runtime、交易語氣或使用者理解宣稱時，必須升級到對應測試或人工 review。
3. 定期複檢：每次完成重大變更後，複檢契約矩陣是否仍符合核心目標、使用者決策用途、維護者合併判斷與契約安全邊界。
4. 不得把第 10 次收尾解讀為 runtime 安全或使用者理解已驗證。
5. 不得新增 runtime、遙測或自動選測工具；本章只收尾合理性、可驗證性與制定策略。

## 報告模板拆解

### `v1` 模式 A：學術深度派

- 使用情境：建立長線研究清單、比較基本面品質、檢查估值與護城河。
- 必要報告重點：資料可信度、財務趨勢、護城河、估值區間、多空辯論、最終投資建議。
- 可驗證性：引用最新價格、財務期間、估值假設與多空證據；若資料不足，要在資料可信度與來源審計揭露。
- 下一步：納入長線研究清單、等待估值區間、或排除。

### `v2` 模式 B：實戰交易派

- 使用情境：已有關注標的，需要決定進場、續抱、減碼或等待。
- 必要報告重點：總經環境、籌碼動向、情緒與技術、估值區間、部位風險。
- 可驗證性：進出場理由要連到資料欄位或事件日期；不得只用主觀語氣給出買賣結論。
- 下一步：建立部位、調整持倉、設定重跑或維持觀望。

### `v3` 模式 C：逆勢交易與泡沫狙擊

- 使用情境：市場敘事過熱、估值偏離、或需要檢查避險與做空風險。
- 必要報告重點：泡沫證據鏈、法證財務、籌碼派發、反證條件、做空觸發與停損。
- 可驗證性：每個泡沫判斷要有對應財務、估值、情緒、籌碼或事件證據；做空建議必須包含失效條件。
- 下一步：避開、避險、等待觸發、或建立反向觀察清單。

### `v4` 模式 D：極短線波段與事件驅動

- 使用情境：1-2 週事件窗口、技術動能、主力籌碼或催化劑交易。
- 必要報告重點：短線方向、進場區間、目標、停損、催化事件、資料新鮮度。
- 可驗證性：所有價格區間、停損與催化日期必須可追溯；若技術、籌碼與事件互相衝突，交易方向應偏中性。
- 下一步：排入短線任務、等待觸發、取消交易或更新 watchlist。

## 連續模式

`both` 代表依序執行 `v1`、`v2`、`v3`，用途是同一檔股票需要長線、交易與逆勢三視角交叉檢查。它不是獨立報告模板；完成後應產出三份獨立報告，並由使用者比較結論是否一致。

## 驗收標準

- 前端模式 label、CTA、hint 與 watchlist 顯示不得自行維護另一份語意表。
- 報告必須顯示對應的摘要標題、決策標題、適用受眾、核心問題與閱讀路徑。
- 報告必須顯示對應的決策紀律標題；Mode B/C/D 不得退回通用「投資論文與決策紀律」。
- 模式 A/B/C/D 的報告模板不可共用同一套決策標題或決策紀律語境。
- 任何新增模式都必須補齊 pipeline definition、mode template profile、前端 metadata、報告模板測試與本契約。
